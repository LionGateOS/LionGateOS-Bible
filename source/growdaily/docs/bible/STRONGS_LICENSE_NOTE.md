# Strong's Data Source Note

Audit date: 2026-06-07

Files reviewed:
- `assets/strong/strong_kjv.json`
- `assets/strong/strong_dict_hebrew.json`
- `assets/strong/strong_dict_greek.json`
- `scripts/buildStrongsMapping.js`
- `scripts/buildStrongsDict.js`
- `scripts/strongs-hebrew-dictionary.js`
- `scripts/strongs-greek-dictionary.js`

Confirmed locally:
- `strong_kjv.json` is generated from `https://github.com/kaiserlik/kjv` per `scripts/buildStrongsMapping.js` comments and the bundled `meta.source` field.
- The bundled mapping is a source-order Strong's ID sequence per verse slot. It is not documented locally as a guaranteed rendered-English-word index.
- `strong_dict_hebrew.json` and `strong_dict_greek.json` are generated by `scripts/buildStrongsDict.js`.
- `scripts/buildStrongsDict.js` points to the Open Scriptures Strong's sources and labels them `public domain / CC-BY-SA`.
- The bundled source JS headers say:
  - Hebrew: `Copyright 2010, Open Scriptures. CC-BY-SA. Derived from XML.`
  - Greek: `Copyright 2009, Open Scriptures. CC-BY-SA. Derived from XML.`
- The Hebrew source header also states the XML e-text was cleaned from several public-domain sources.

Still worth reviewing:
- The generated JSON files in `assets/strong/` do not currently embed attribution/license metadata of their own.
- App/package distribution should keep appropriate attribution for the Open Scriptures source files and review any CC-BY-SA redistribution obligations for bundled derived data.

Current safety rule:
- GrowDaily may display bundled Strong's numbers, lemma, transliteration, gloss, and KJV-definition fields with attribution/context.
- UI should continue to note that exact rendered English-word placement is not confirmed by the current mapping source.
- If the Strong's dictionary pipeline changes, re-check attribution and redistribution requirements before replacing the bundled files.
